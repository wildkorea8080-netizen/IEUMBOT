# 17. 로깅 및 모니터링 명세

이 문서는 이음봇(IEUMBOT)의 로깅, 모니터링, 관측성 요구사항을 정의합니다. 목표는 장애를 빠르게 탐지하고, 원인을 추적하며, 품질 저하를 조기에 감지하는 것입니다.

## 1. 애플리케이션 로그

### 1.1 로그 원칙

- 구조화 JSON 로그를 기본으로 사용
- 모든 요청에 `requestId` 부여
- `organizationId`, `chatbotId`를 가능한 범위에서 포함
- 로그 레벨 표준화: `DEBUG`, `INFO`, `WARN`, `ERROR`
- 타임스탬프는 UTC 기준 `ISO-8601` 사용

### 1.2 필수 로그 필드

| 필드 | 설명 |
|---|---|
| `timestamp` | 로그 시각 |
| `level` | 로그 레벨 |
| `service` | 서비스 이름(`api`, `worker`, `web`) |
| `environment` | `local`, `staging`, `prod` |
| `requestId` | 요청 추적 ID |
| `organizationId` | 조직 ID |
| `chatbotId` | 챗봇 ID |
| `route` | API 경로 |
| `method` | HTTP 메서드 |
| `statusCode` | 응답 코드 |
| `latencyMs` | 응답 시간(ms) |
| `errorCode` | 오류 코드(있을 때) |

### 1.3 무엇을 로그로 남길지

필수:

- API 요청 시작/종료
- 인증 실패/권한 실패
- 채팅 요청 상태(`completed`, `limited`, `failed`)
- 모델 호출 결과(성공/실패, 지연 시간, 토큰 사용량)
- 문서 인입 단계별 상태 변화
- SSE 연결 시작/종료/오류

예시(JSON):

```json
{
  "timestamp": "2026-04-23T08:00:00Z",
  "level": "INFO",
  "service": "api",
  "requestId": "req_abc123",
  "organizationId": "org_1",
  "chatbotId": "cb_1",
  "route": "/api/chat/messages/stream",
  "method": "POST",
  "statusCode": 200,
  "latencyMs": 3200,
  "answerStatus": "completed",
  "citationCount": 2
}
```

### 1.4 무엇을 로그로 남기지 말아야 하는지

금지:

- OpenAI API 키, DB 비밀번호, 세션 시크릿
- 개인정보 원문(전화번호, 이메일, 주민번호 등)
- 마스킹 전 사용자 질문 원문
- 내부 시스템 프롬프트 전체 원문
- 쿠키/토큰 원문

예시(금지):

- `Authorization` 헤더 원문
- `Set-Cookie` 값
- `contentOriginal` 원문

## 2. 감사 로그

### 2.1 목적

- 관리자 행위의 추적 가능성 확보
- 사후 감사, 권한 오남용 조사, 변경 이력 복원 지원

### 2.2 감사 이벤트 대상

- 로그인 성공/실패/로그아웃
- 챗봇 설정 변경
- 문서 업로드/삭제/비활성화/재처리
- 개인정보 정책 변경
- 관리자 역할 변경
- 웹 소스/퀵액션 변경

### 2.3 감사 로그 필드

| 필드 | 설명 |
|---|---|
| `auditId` | 감사 로그 ID |
| `adminId` | 수행 관리자 ID |
| `organizationId` | 조직 ID |
| `action` | 행위 코드(`document.upload`) |
| `targetType` | 대상 타입 |
| `targetId` | 대상 ID |
| `result` | `success`, `failure` |
| `requestId` | 요청 ID |
| `createdAt` | 생성 시각 |

규칙:

- 감사 로그 수정/삭제 API 미제공
- 고권한 사용자만 조회 가능

## 3. API 메트릭

### 3.1 공통 API 메트릭

- 요청 수(`http_requests_total`)
- 오류 수(`http_errors_total`)
- 응답 시간(`http_request_duration_ms`)
- 상태 코드 분포(2xx/4xx/5xx)
- 엔드포인트별 P50/P95/P99 latency

### 3.2 핵심 API 목표(초안)

| 항목 | 목표 |
|---|---|
| 일반 API 평균 응답시간 | 2초 이내 |
| 일반 API P95 | 5초 이내 |
| 채팅 API P95 | 15초 이내 |
| 5xx 오류율(5분) | 1% 이하 |

## 4. 챗봇 품질 메트릭

핵심 품질 지표:

- 답변 성공률(`answer_completed_rate`)
- 제한 응답 비율(`answer_limited_rate`)
- 답변 실패율(`answer_failed_rate`)
- 출처 포함 응답 비율(`citation_attached_rate`)
- 평균 응답 시간
- 재질문율(동일 세션 연속 질문 비율)

운영 목표 예:

| 지표 | 목표 |
|---|---|
| 출처 포함 응답 비율 | 80% 이상 |
| 제한 응답 비율 | 30% 이하(도메인별 조정) |
| 답변 실패율 | 3% 이하 |

주의:

- 제한 응답 비율은 무조건 낮을수록 좋은 지표가 아님
- 근거 없는 답변을 줄이는 정책과 함께 해석

## 5. 문서 처리 메트릭

수집 지표:

- 업로드 건수
- 인입 성공률
- 인입 실패율
- 단계별 실패율(`extract`, `chunk`, `embed`, `index`)
- 처리 소요 시간(문서 크기별)
- 재시도 횟수

운영 목표 예:

| 지표 | 목표 |
|---|---|
| 인입 성공률 | 95% 이상 |
| 50p PDF 처리 시간 | 5분 이내 |
| 200p PDF 처리 시간 | 15분 이내 |

실패 분석 필수 항목:

- `errorCode`
- `currentStep`
- `attemptCount`
- `documentVersionId`

## 6. 인프라 메트릭

대상:

- API 서버
- 워커
- PostgreSQL
- Redis
- Nginx

필수 메트릭:

- CPU 사용률
- 메모리 사용률
- 디스크 사용률
- 네트워크 I/O
- 프로세스 재시작 횟수
- DB 커넥션 수
- Redis 메모리/큐 길이
- Nginx 5xx 비율

권장 기준(초안):

| 메트릭 | 경고 임계치 |
|---|---|
| API CPU | 80% 이상 10분 지속 |
| API 메모리 | 85% 이상 10분 지속 |
| DB 커넥션 사용률 | 80% 이상 5분 지속 |
| Redis 큐 적체 | 1000건 이상 10분 지속 |
| 디스크 사용률 | 85% 이상 |

## 7. 알림 기준

알림은 `INFO`가 아닌 실질 장애 가능성 중심으로 설계합니다.

필수 알림(초안):

1. API 5xx 오류율(5분 평균) > 5%
2. 채팅 API P95 latency > 15초 (10분 지속)
3. 모델 호출 실패율(5분) > 10%
4. 문서 인입 실패율(1시간) > 20%
5. Redis 큐 적체 > 1000 (10분)
6. DB 연결 실패 발생
7. Nginx upstream 연결 실패 급증

알림 메시지 필수 정보:

- 서비스명
- 환경
- 지표명
- 현재값/임계치
- 발생 시각
- 대응 런북 링크

## 8. 대시보드 항목

### 8.1 운영 대시보드

- 분당 요청 수
- 2xx/4xx/5xx 비율
- API latency(P50/P95/P99)
- 채팅 성공/제한/실패율
- 모델 호출 성공률
- SSE 활성 연결 수

### 8.2 문서 처리 대시보드

- 시간대별 업로드 건수
- 인입 성공/실패 건수
- 단계별 실패 분포
- 평균 처리 시간
- 큐 길이/지연 시간

### 8.3 보안/감사 대시보드

- 로그인 실패 횟수
- 권한 거부 횟수(403)
- 관리자 설정 변경 이벤트 수
- 개인정보 마스킹 탐지 건수

## 9. 장애 추적 방식

### 9.1 추적 키

- `requestId`를 API, 워커, 모델 호출, 로그 저장 단계에 공통 사용
- 문서 처리에는 `jobId` 추가 사용

### 9.2 장애 분석 흐름

1. 알림 수신
2. 대시보드에서 서비스/시간대 확인
3. `requestId` 또는 `jobId`로 연관 로그 조회
4. 오류 코드와 단계(`currentStep`) 확인
5. 영향 범위(조직/챗봇/문서) 산정
6. 임시 완화(재시도, 롤백, 제한 응답 강화)
7. 근본 원인 분석 및 재발 방지

### 9.3 필수 운영 규칙

- 장애 티켓에는 최소 1개 이상 `requestId` 첨부
- 장애 종료 후 원인/대응/재발방지 항목 기록

## 10. 로그 보존 정책

로그 유형별 권장 보존 기간:

| 로그 유형 | 권장 보존 |
|---|---|
| 애플리케이션 로그 | 30~90일 |
| 채팅 로그(마스킹) | 90일 기본 |
| 오류 로그 | 90~180일 |
| 감사 로그 | 1년 이상(기관 정책 우선) |
| 인입 작업 로그 | 90일 |

운영 규칙:

- 보존 기간 경과 데이터는 삭제 또는 익명화
- 정책 변경은 감사 로그에 기록
- 백업 로그도 동일한 접근통제 정책 적용

## 11. 구현 체크리스트

- 모든 API 응답에 `requestId`가 연결된다.
- 개인정보 원문이 로그에 남지 않는다.
- 채팅/인입/감사 로그가 분리 저장된다.
- 주요 지표(P95 latency, error rate, ingestion failure)가 수집된다.
- 알림 임계치가 설정되고 검증되었다.
- 장애 런북 링크가 알림 메시지에 포함된다.
- 대시보드가 운영/인입/보안 관점으로 분리되어 있다.
