# 06. 비기능 요구사항

이 문서는 이음봇(IEUMBOT)의 공개 위젯, 관리자 콘솔, 문서 처리, RAG 질의응답, 운영 시스템에 적용할 비기능 요구사항을 정의합니다. MVP에서 현실적으로 달성 가능한 기준을 우선하고, 운영 환경으로 확장 가능한 목표를 함께 제시합니다.

## 1. 성능

### 위젯 로딩

- 위젯 스크립트 초기 로드 목표: 일반 4G 또는 사무실 네트워크 기준 2초 이내
- 런처 버튼 표시 목표: 설정 API 응답 포함 2초 이내
- 위젯 번들 크기 목표:
  - MVP: gzip 기준 250KB 이하 권장
  - 운영 확장: 코드 분할과 지연 로딩 적용
- 호스트 웹사이트의 주요 렌더링을 차단하지 않아야 합니다.

### 채팅 응답

- 사용자 질문 후 첫 응답 표시 목표:
  - 일반 문서 기반 답변: 8초 이내
  - 복잡한 검색 또는 모델 지연 상황: 15초 이내
- 15초를 초과할 가능성이 있으면 로딩 상태를 유지하고 사용자에게 처리 중임을 표시합니다.
- 30초 초과 시 타임아웃으로 처리하고 재시도 또는 대체 안내를 제공합니다.

### 관리자 콘솔

- 주요 목록 화면 로딩 목표: 2초 이내
- 대시보드 요약 지표 로딩 목표: 3초 이내
- 저장 요청 응답 목표: 2초 이내
- 로그 검색은 기간 필터 기준 5초 이내 응답을 목표로 합니다.

### 문서 처리

- 일반 텍스트 PDF 50페이지 기준 처리 목표: 5분 이내
- 일반 텍스트 PDF 200페이지 기준 처리 목표: 15분 이내
- 처리 단계별 상태를 저장해 장시간 작업 중에도 진행 상태를 확인할 수 있어야 합니다.
- 스캔 PDF OCR은 MVP 범위 밖이며, 향후 별도 성능 목표를 정의합니다.

## 2. 가용성

### 서비스 가용성 목표

- MVP 목표 가용성: 월 99.0% 이상
- 운영 안정화 이후 목표 가용성: 월 99.5% 이상
- 주요 대상:
  - 공개 위젯 설정 API
  - 채팅 API
  - 관리자 콘솔
  - 문서 처리 큐와 워커

### 장애 허용 기준

- 문서 처리 워커 장애가 발생해도 기존 활성 문서 기반 채팅은 계속 동작해야 합니다.
- 새 문서 인입 실패가 기존 답변 품질에 영향을 주지 않아야 합니다.
- 관리자 콘솔 장애가 공개 위젯 채팅 장애로 전파되지 않도록 구성합니다.

### 점검 시간

- MVP에서는 정기 점검 시간을 사전 공지할 수 있습니다.
- 운영 환경에서는 가능한 무중단 배포를 목표로 합니다.
- 점검 중에는 관리자 콘솔에 점검 안내를 표시하고, 공개 위젯은 가능한 기존 답변 기능을 유지합니다.

## 3. 확장성

### 트래픽 확장

- MVP 기준:
  - 단일 조직 또는 제한된 조직 수 지원
  - 동시 채팅 요청 수 50건 수준을 안정적으로 처리
  - 일 5,000개 사용자 메시지 처리 가능
- 운영 확장 목표:
  - 조직별 설정, 문서 색인, 로그를 분리해 멀티 테넌트 확장이 가능해야 합니다.
  - API 서버와 문서 처리 워커는 수평 확장 가능한 구조를 유지합니다.

### 데이터 확장

- MVP 기준:
  - 조직당 PDF 100개
  - 조직당 총 문서 페이지 10,000페이지
  - 조직당 채팅 로그 100만 건까지 보관 정책 내 관리 가능
- 운영 확장:
  - 로그와 분석 데이터는 운영 DB와 분리 가능한 구조를 고려합니다.
  - 벡터 색인은 조직, 문서, 버전 기준 필터링이 가능해야 합니다.

### 작업 큐 확장

- 문서 처리 작업은 큐 기반으로 처리합니다.
- 대량 업로드 시 API 요청은 빠르게 종료하고, 처리 상태는 비동기로 갱신합니다.
- 워커 수를 늘려 문서 처리량을 확장할 수 있어야 합니다.

## 4. 보안

### 인증과 권한

- 관리자 콘솔은 인증된 사용자만 접근할 수 있어야 합니다.
- 모든 관리자 API는 역할 기반 권한 검사를 수행해야 합니다.
- 권한 없는 요청은 `403`을 반환하고, 세션 없는 요청은 `401`을 반환합니다.
- 관리자 로그인 성공, 실패, 로그아웃, 권한 변경은 감사 로그에 기록합니다.

### API 보안

- 모든 운영 환경 통신은 HTTPS를 사용해야 합니다.
- 위젯 설정 요청은 허용 도메인 검증을 수행해야 합니다.
- API 응답에는 내부 오류 스택, 비밀 키, 시스템 경로를 포함하지 않습니다.
- 요청 크기 제한과 속도 제한을 적용해야 합니다.

### 파일 업로드 보안

- 업로드 파일은 허용된 확장자와 MIME 타입을 검증합니다.
- PDF 최대 크기 제한을 적용합니다.
- 파일명은 저장 시 안전한 내부 식별자로 변환합니다.
- 원본 파일 접근은 인증과 권한 확인을 거쳐야 합니다.

### 비밀 관리

- 모델 API 키, 데이터베이스 접속 정보, 스토리지 키는 코드에 저장하지 않습니다.
- 환경 변수 또는 비밀 관리 도구를 사용합니다.
- 운영 로그에 비밀 값이 출력되지 않아야 합니다.

## 5. 개인정보보호

### 수집 최소화

- 일반 사용자 채팅은 기본적으로 익명 세션으로 처리합니다.
- 세션 ID에는 이름, 이메일, 전화번호 등 직접 식별자를 포함하지 않습니다.
- 서비스 제공에 필요하지 않은 개인정보 입력을 유도하지 않습니다.

### 마스킹

- 이메일, 전화번호, 주민등록번호 패턴은 로그 저장 전 마스킹합니다.
- 관리자 화면에는 마스킹된 질문과 답변만 표시합니다.
- 개인정보 원문 복구 기능은 MVP에서 제공하지 않습니다.
- 기본 개인정보 패턴 테스트셋 기준 마스킹 적용률 95% 이상을 목표로 합니다.

### 보관과 삭제

- 채팅 로그 기본 보관 기간은 MVP에서 90일을 권장합니다.
- 운영 정책에 따라 30일, 90일, 180일 등으로 조정 가능해야 합니다.
- 보관 기간이 지난 로그는 삭제 또는 익명화 대상이 됩니다.
- 삭제 정책 실행 결과는 운영 로그에 기록합니다.

### 모델 호출

- 모델 호출 전 개인정보 마스킹 적용을 우선합니다.
- 모델 입력에는 답변 생성에 필요한 최소 문맥만 포함합니다.
- 모델 응답은 저장 전 정책 검사를 거쳐야 합니다.

## 6. 유지보수성

### 코드 구조

- 위젯, 관리자 콘솔, API 서버, 문서 처리 워커, RAG 계층은 책임을 분리합니다.
- 공통 타입과 API 계약은 중복 정의를 최소화합니다.
- 문서 처리 파이프라인은 단계별로 테스트 가능해야 합니다.

### 설정 관리

- 조직별 챗봇 설정은 데이터로 관리하고 코드 배포 없이 변경 가능해야 합니다.
- 환경별 설정은 개발, 스테이징, 운영으로 분리합니다.
- 설정 변경은 버전과 변경 이력을 가져야 합니다.

### 문서화

- API, 데이터 모델, 배포 절차, 운영 절차는 `docs/`에 유지합니다.
- 기능 변경 시 관련 문서를 함께 갱신합니다.
- 주요 설계 결정은 변경 이력 또는 별도 설계 문서에 기록합니다.

### 테스트 가능성

- 핵심 도메인 로직은 단위 테스트 가능해야 합니다.
- RAG 검색, 마스킹, 권한, 문서 처리 단계는 독립 테스트를 제공해야 합니다.
- 테스트 데이터는 실제 개인정보를 포함하지 않아야 합니다.

## 7. 관측성

### 로깅

- 모든 채팅 요청에는 `requestId`를 부여합니다.
- 위젯, API, RAG, 모델 호출, 로그 저장 단계가 같은 `requestId`로 추적 가능해야 합니다.
- 로그에는 최소한 다음 항목을 포함합니다.
  - 요청 ID
  - 조직 ID 또는 챗봇 ID
  - 세션 ID
  - 처리 상태
  - 응답 시간
  - 오류 코드
  - 출처 문서 ID

### 지표

- MVP에서 수집할 핵심 지표:
  - 총 대화 수
  - 평균 응답 시간
  - P95 응답 시간
  - 답변 실패율
  - 출처 포함 응답 비율
  - 문서 처리 성공률
  - 모델 호출 실패율
  - 개인정보 마스킹 탐지 건수

### 알림

- MVP에서는 관리자 대시보드와 오류 로그 기반 확인을 우선합니다.
- 운영 확장 시 다음 조건에 대해 알림을 설정합니다.
  - 채팅 API 오류율 5분 평균 5% 초과
  - 모델 호출 실패율 5분 평균 10% 초과
  - 문서 처리 실패율 1시간 평균 20% 초과
  - API P95 응답 시간 15초 초과

### 감사 추적

- 관리자 설정 변경, 문서 변경, 권한 변경은 감사 로그에 기록합니다.
- 감사 로그는 일반 관리자 수정 기능을 제공하지 않습니다.
- 감사 로그 보관 기간은 운영 정책에 따라 별도로 정의합니다.

## 8. 접근성

### 위젯 접근성

- 런처 버튼은 키보드 포커스와 접근성 라벨을 제공해야 합니다.
- 채팅 패널은 키보드로 열고 닫을 수 있어야 합니다.
- 입력창, 전송 버튼, 닫기 버튼은 명확한 포커스 상태를 가져야 합니다.
- 메시지 로딩 상태는 스크린 리더가 인지할 수 있는 방식으로 제공해야 합니다.

### 시각 접근성

- 텍스트와 배경의 색상 대비는 WCAG 2.1 AA 수준을 목표로 합니다.
- 버튼 텍스트는 모바일에서도 잘리지 않아야 합니다.
- 오류 상태는 색상만으로 구분하지 않고 텍스트 또는 아이콘을 함께 사용합니다.

### 관리자 콘솔 접근성

- 폼 필드는 라벨과 오류 메시지를 명확히 연결합니다.
- 테이블은 헤더와 셀 관계를 유지합니다.
- 모달은 포커스 트랩과 닫기 동작을 지원합니다.

## 9. 호환성

### 브라우저 호환성

- MVP 지원 브라우저:
  - Chrome 최신 2개 주요 버전
  - Microsoft Edge 최신 2개 주요 버전
  - Safari 최신 2개 주요 버전
  - Firefox 최신 2개 주요 버전
- Internet Explorer는 지원하지 않습니다.

### 모바일 호환성

- iOS Safari와 Android Chrome 최신 주요 버전을 지원합니다.
- 최소 화면 너비 360px에서 위젯 주요 기능이 동작해야 합니다.
- 모바일에서 채팅 패널은 화면을 벗어나지 않아야 합니다.

### 호스트 웹사이트 호환성

- 위젯은 일반 HTML 사이트, CMS 기반 사이트, SPA에 삽입 가능해야 합니다.
- 위젯 스크립트는 호스트 사이트 전역 CSS와 충돌하지 않아야 합니다.
- 위젯은 호스트 사이트의 라우팅과 폼 동작을 방해하지 않아야 합니다.

### 파일 호환성

- MVP 문서 지원 형식은 PDF입니다.
- 텍스트 기반 PDF를 우선 지원합니다.
- 암호화 PDF, 손상 PDF, 스캔 PDF는 제한 또는 실패 상태로 처리합니다.

## 10. 운영성

### 관리자 운영

- 운영자는 관리자 콘솔에서 문서 상태, 최근 오류, 채팅 로그를 확인할 수 있어야 합니다.
- 문서 처리 실패 시 실패 단계와 재시도 버튼을 제공해야 합니다.
- 설정 변경은 저장 직후 확인 가능해야 하며, 위젯 반영 지연이 있으면 안내해야 합니다.

### 환경 관리

- 개발, 스테이징, 운영 환경을 분리합니다.
- 운영 데이터는 개발 환경으로 직접 복제하지 않습니다.
- 스테이징에서는 운영과 유사한 배포 절차를 검증합니다.

### 운영 정책

- 로그 보관 기간, 업로드 파일 크기, 세션 만료 시간, 요청 제한은 설정값으로 관리합니다.
- 운영 중 변경 가능한 설정과 배포가 필요한 설정을 구분합니다.
- 정책 변경은 관리자와 개발자가 추적할 수 있어야 합니다.

### 지원과 장애 대응

- 오류 화면과 로그에는 요청 ID를 표시하거나 복사할 수 있어야 합니다.
- 장애 문의 시 요청 ID로 관련 로그를 추적할 수 있어야 합니다.
- 자주 발생하는 오류는 운영 가이드에 대응 절차를 기록합니다.

## 11. 배포 안정성

### 배포 방식

- API 서버, 관리자 콘솔, 위젯, 문서 처리 워커는 독립 배포 가능한 구조를 목표로 합니다.
- MVP에서는 단순 배포를 허용하되, 역할 분리는 코드와 설정에서 유지합니다.
- 배포 전 스테이징 환경에서 기본 회귀 테스트를 수행합니다.

### 마이그레이션

- 데이터베이스 스키마 변경은 마이그레이션 스크립트로 관리합니다.
- 운영 배포 전 마이그레이션 롤백 가능성을 검토합니다.
- 파괴적 변경은 사전 백업 후 진행합니다.

### 롤백

- 애플리케이션 배포는 이전 안정 버전으로 롤백 가능해야 합니다.
- 위젯 배포는 캐시 영향을 고려해 버전 경로 또는 파일명 해시를 사용합니다.
- 설정 변경은 이전 버전으로 되돌릴 수 있도록 변경 이력을 유지합니다.

### 배포 검증

- 배포 후 최소 검증 항목:
  - 위젯 설정 로드
  - 테스트 질문 응답
  - 관리자 로그인
  - 문서 목록 조회
  - 로그 기록 확인
- 배포 후 오류율과 응답 시간을 일정 시간 모니터링합니다.

## 12. 복구 전략

### 백업

- 운영 데이터베이스는 최소 일 1회 백업합니다.
- 문서 원본 파일과 색인 데이터는 복구 가능한 저장소에 보관합니다.
- 백업 보관 기간은 MVP에서 최소 7일을 권장합니다.
- 운영 확장 시 30일 이상 보관을 검토합니다.

### 복구 목표

- MVP 복구 목표:
  - RPO: 24시간 이내
  - RTO: 4시간 이내
- 운영 확장 목표:
  - RPO: 1시간 이내
  - RTO: 1시간 이내

### 장애별 복구

- API 서버 장애:
  - 이전 안정 버전 재배포 또는 인스턴스 재시작
  - 장애 시간 동안 요청 실패 로그 확인

- 문서 처리 워커 장애:
  - 워커 재시작 후 실패 작업 재시도
  - 기존 활성 색인은 유지

- 벡터 색인 장애:
  - 최신 문서 조각과 임베딩 메타데이터로 색인 재생성
  - 재생성 중에는 제한 응답 또는 기존 색인 사용 정책 적용

- 데이터베이스 장애:
  - 최신 백업에서 복구
  - 복구 후 문서 색인과 로그 정합성 점검

### 복구 훈련

- 운영 전 최소 1회 백업 복구 절차를 검증합니다.
- 운영 안정화 이후 분기 1회 복구 리허설을 권장합니다.
- 복구 절차와 소요 시간은 운영 문서에 기록합니다.
