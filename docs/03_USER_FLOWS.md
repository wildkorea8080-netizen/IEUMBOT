# 03. 사용자 흐름

이 문서는 이음봇(IEUMBOT)의 일반 사용자 위젯 흐름과 관리자 콘솔 흐름을 화면, 상태, API, 예외 처리 기준으로 정리합니다.

## 1. 일반 사용자 상담 시작 플로우

### 시작 조건

- 사용자가 이음봇 위젯이 삽입된 웹사이트에 접속합니다.
- 브라우저에서 위젯 스크립트가 정상 로드됩니다.
- 현재 도메인이 관리자 콘솔에 등록된 허용 도메인입니다.
- 챗봇 설정이 활성 상태입니다.

### 단계별 흐름

1. 위젯 스크립트가 실행됩니다.
2. 위젯은 공개 설정 키 또는 챗봇 ID로 `GET /api/widget/config` 요청을 보냅니다.
3. 서버는 챗봇 이름, 인사말, 테마, 빠른 액션, 운영시간, 개인정보 안내 문구를 반환합니다.
4. 위젯은 화면 하단에 플로팅 아이콘을 표시합니다.
5. 사용자가 플로팅 아이콘을 클릭합니다.
6. 위젯은 채팅 패널을 엽니다.
7. 최초 세션이면 익명 세션 ID를 생성하거나 서버에서 발급받습니다.
8. 채팅 패널 상단에 챗봇 이름과 닫기 버튼을 표시합니다.
9. 본문 영역에 인사말, 개인정보 입력 주의 문구, 빠른 액션 버튼을 표시합니다.
10. 입력창은 질문 입력 가능 상태로 표시합니다.

### 예외 상황

- 위젯 설정 로드 실패
  - 플로팅 아이콘을 숨기거나 기본 오류 상태를 표시합니다.
  - 콘솔 오류에는 설정 요청 실패 원인을 남깁니다.

- 허용되지 않은 도메인
  - 서버는 `403`을 반환합니다.
  - 위젯은 사용자에게 노출되지 않습니다.

- 챗봇 비활성 상태
  - 서버는 비활성 상태와 안내 문구를 반환합니다.
  - 위젯은 숨김 또는 "현재 상담을 사용할 수 없습니다" 상태를 표시합니다.

- 네트워크 지연
  - 설정 로드 중에는 아이콘을 늦게 표시하거나 로딩 상태를 최소화합니다.
  - 5초 이상 실패하면 재시도 없이 조용히 종료합니다.

### 성공 조건

- 사용자는 플로팅 아이콘을 클릭해 채팅 패널을 열 수 있습니다.
- 채팅 패널에는 챗봇 이름, 인사말, 입력창, 빠른 액션이 표시됩니다.
- 세션 ID가 생성되어 이후 질문 요청에 연결됩니다.

## 2. 일반 사용자 질문-응답 플로우

### 시작 조건

- 사용자가 채팅 패널을 열었습니다.
- 입력창이 활성화되어 있습니다.
- 챗봇과 연결된 문서 색인이 사용 가능한 상태입니다.

### 단계별 흐름

1. 사용자가 텍스트 질문을 입력합니다.
2. 사용자가 전송 버튼을 클릭하거나 Enter 키로 질문을 전송합니다.
3. 위젯은 사용자 메시지를 대화 목록에 즉시 표시합니다.
4. 입력창은 전송 중 상태로 전환됩니다.
5. 위젯은 `POST /api/chat/messages` 요청을 보냅니다.
6. 요청에는 챗봇 ID, 세션 ID, 사용자 질문, 현재 대화 컨텍스트 ID를 포함합니다.
7. 서버는 질문에서 개인정보 패턴을 탐지하고 마스킹 정책을 적용합니다.
8. 서버는 질문 임베딩을 생성하고 관련 문서 조각을 검색합니다.
9. 검색 결과가 충분하면 RAG 프롬프트를 구성합니다.
10. 모델은 문서 근거 기반 답변을 생성합니다.
11. 서버는 답변, 출처 목록, 응답 상태, 요청 ID를 반환합니다.
12. 위젯은 답변을 메시지 목록에 표시합니다.
13. 출처가 있으면 문서명, 페이지, 링크 또는 펼침 버튼을 표시합니다.
14. 입력창은 다시 활성화됩니다.

### 예외 상황

- 빈 질문
  - 전송하지 않고 입력창 하단에 짧은 안내를 표시합니다.

- 질문 길이 초과
  - 전송 전에 제한 문구를 표시합니다.
  - 서버에서도 동일한 제한을 검증합니다.

- 개인정보 포함
  - 서버는 로그 저장 전 마스킹합니다.
  - 필요하면 답변에 개인정보 입력 자제 안내를 포함합니다.

- 근거 문서 부족
  - 서버는 제한 응답을 반환합니다.
  - 위젯은 "등록된 문서에서 확인할 수 없습니다" 형식의 답변과 대체 문의 안내를 표시합니다.

- 모델 호출 실패
  - 서버는 사용자용 오류 메시지를 반환합니다.
  - 내부 오류 상세는 로그에만 저장합니다.

- 네트워크 오류
  - 위젯은 실패 메시지와 재시도 버튼을 표시합니다.
  - 재시도 시 같은 질문과 세션 ID를 다시 전송합니다.

### 성공 조건

- 사용자는 질문에 대한 답변을 확인합니다.
- 답변은 검색된 문서 근거를 기반으로 생성됩니다.
- 가능한 경우 출처 정보가 함께 표시됩니다.
- 요청과 응답은 로그 추적 ID로 연결됩니다.

## 3. 일반 사용자 재질문/연속 대화 플로우

### 시작 조건

- 사용자가 이전 질문과 답변을 받은 상태입니다.
- 세션 ID와 대화 컨텍스트가 유지되고 있습니다.

### 단계별 흐름

1. 사용자가 이전 답변을 보고 후속 질문을 입력합니다.
2. 예: "그럼 제출 서류는 뭐가 필요해요?"
3. 위젯은 기존 세션 ID와 최근 대화 ID를 포함해 `POST /api/chat/messages`를 호출합니다.
4. 서버는 후속 질문이 이전 질문을 참조하는지 판단합니다.
5. 필요한 경우 최근 대화 요약 또는 직전 검색 결과를 질의 재작성에 사용합니다.
6. 서버는 재작성된 검색 질의로 문서 조각을 검색합니다.
7. 검색 결과와 대화 맥락을 함께 사용해 답변을 생성합니다.
8. 위젯은 답변과 출처를 기존 대화 아래에 추가합니다.
9. 대화가 길어지면 서버는 컨텍스트 길이 제한에 맞춰 요약된 맥락만 유지합니다.

### 예외 상황

- 대화 세션 만료
  - 서버는 세션 만료 상태를 반환합니다.
  - 위젯은 새 상담으로 이어가도록 안내합니다.

- 후속 질문의 참조 대상 불명확
  - 챗봇은 추가 확인 질문을 반환합니다.
  - 예: "어떤 사업의 제출 서류를 말씀하시는지 알려주세요."

- 이전 답변이 근거 부족 상태였음
  - 서버는 이전 근거를 확장하지 않고 새 검색을 수행합니다.
  - 충분한 근거가 없으면 다시 제한 응답을 반환합니다.

- 컨텍스트 길이 초과
  - 오래된 대화는 요약하거나 제외합니다.
  - 사용자가 볼 수 있는 대화 목록은 유지하되 모델 입력에는 제한적으로 포함합니다.

### 성공 조건

- 사용자는 같은 채팅 세션 안에서 후속 질문을 이어갈 수 있습니다.
- 서버는 필요한 대화 맥락을 활용하되 문서 근거 기반 응답 원칙을 유지합니다.
- 불명확한 질문은 임의 해석하지 않고 확인 질문으로 처리됩니다.

## 4. 운영시간 외 안내 플로우

### 시작 조건

- 챗봇 설정에 운영시간이 등록되어 있습니다.
- 현재 시간이 운영시간 외입니다.
- 챗봇 자체 자동응답은 활성화되어 있거나, 상담원 연결 기능은 비활성 또는 미제공 상태입니다.

### 단계별 흐름

1. 위젯은 설정 로드 시 운영시간 정보를 받습니다.
2. 사용자가 운영시간 외에 채팅 패널을 엽니다.
3. 패널 상단 또는 인사말 영역에 운영시간 외 안내를 표시합니다.
4. 사용자는 문서 기반 자동응답 질문을 입력할 수 있습니다.
5. 질문이 접수되면 일반 질문-응답 플로우와 동일하게 처리합니다.
6. 담당자 문의가 필요한 답변의 경우 운영시간, 문의처, 다음 영업일 안내를 함께 표시합니다.
7. 상담원 연결 기능이 향후 제공되는 경우, 연결 버튼은 비활성 상태 또는 예약 안내 상태로 표시합니다.

### 예외 상황

- 운영시간 설정 없음
  - 운영시간 외 배너를 표시하지 않습니다.
  - 자동응답은 정상 제공됩니다.

- 서버와 클라이언트 시간 차이
  - 운영시간 판단은 서버 시간을 기준으로 합니다.
  - 위젯은 서버가 반환한 상태를 표시합니다.

- 긴급 공지 설정
  - 운영시간보다 긴급 공지 배너를 우선 표시합니다.
  - 긴급 공지는 관리자 설정에서 별도 제어합니다.

### 성공 조건

- 사용자는 운영시간 외 상태를 명확히 인지합니다.
- 자동응답 가능 범위와 담당자 문의 가능 시간이 구분되어 표시됩니다.
- 운영시간 외에도 문서 기반 Q&A는 정상 동작합니다.

## 5. 관리자 로그인 플로우

### 시작 조건

- 관리자가 콘솔 URL에 접속합니다.
- 관리자 계정이 생성되어 있습니다.
- 인증 서버와 데이터베이스가 정상 동작합니다.

### 단계별 흐름

1. 관리자가 로그인 페이지에 접속합니다.
2. 이메일 또는 계정 ID와 비밀번호를 입력합니다.
3. 로그인 버튼을 클릭합니다.
4. 클라이언트는 `POST /api/admin/auth/login` 요청을 보냅니다.
5. 서버는 계정 존재 여부, 비밀번호, 계정 상태를 검증합니다.
6. 필요 시 2단계 인증 또는 추가 보안 검사를 수행합니다.
7. 인증 성공 시 서버는 세션 쿠키 또는 토큰을 발급합니다.
8. 클라이언트는 관리자 대시보드로 이동합니다.
9. 서버는 로그인 성공 감사 로그를 남깁니다.

### 예외 상황

- 계정 정보 불일치
  - 동일한 오류 문구를 표시해 계정 존재 여부가 노출되지 않게 합니다.

- 계정 비활성화
  - 관리자에게 접근 불가 안내를 표시합니다.
  - 내부 로그에는 비활성 계정 로그인 시도를 기록합니다.

- 반복 실패
  - 일정 횟수 이상 실패 시 임시 잠금 또는 CAPTCHA를 적용합니다.

- 세션 만료
  - 보호된 API가 `401`을 반환합니다.
  - 클라이언트는 로그인 페이지로 이동하고 재로그인을 안내합니다.

### 성공 조건

- 인증된 관리자만 콘솔에 접근합니다.
- 로그인 성공과 실패 이벤트가 감사 로그에 기록됩니다.
- 로그인 후 관리자의 역할에 맞는 메뉴만 표시됩니다.

## 6. 관리자 문서 업로드 플로우

### 시작 조건

- 관리자가 로그인되어 있습니다.
- 관리자는 문서 업로드 권한을 가지고 있습니다.
- 대상 챗봇 또는 조직이 선택되어 있습니다.

### 단계별 흐름

1. 관리자가 문서 관리 화면으로 이동합니다.
2. 업로드 버튼을 클릭합니다.
3. 파일 선택 창 또는 드래그 앤 드롭 영역에 PDF 파일을 추가합니다.
4. 클라이언트는 파일 확장자, 크기, 개수를 1차 검증합니다.
5. 관리자는 문서명, 카테고리, 공개 여부, 메모 등 메타데이터를 입력합니다.
6. 업로드를 실행합니다.
7. 클라이언트는 `POST /api/admin/chatbots/{chatbotId}/documents` 또는 업로드 전용 URL로 파일을 전송합니다.
8. 서버는 파일 형식, 크기, 권한, 악성 파일 가능성을 검증합니다.
9. 서버는 원본 파일을 저장하고 문서 레코드를 `업로드 완료` 상태로 생성합니다.
10. 서버는 문서 처리 작업을 큐에 등록합니다.
11. 콘솔은 문서 상태를 `처리 중`으로 표시합니다.
12. 워커는 텍스트 추출, 청킹, 임베딩, 색인 저장을 수행합니다.
13. 처리 완료 시 상태가 `사용 가능`으로 변경됩니다.
14. 관리자는 테스트 콘솔로 이동해 문서 반영 여부를 확인합니다.

### 예외 상황

- PDF가 아닌 파일
  - 업로드 전 또는 서버 검증 단계에서 차단합니다.
  - 지원 형식 안내를 표시합니다.

- 파일 크기 초과
  - 업로드를 차단하고 최대 크기를 안내합니다.

- 암호화된 PDF
  - 처리 실패 상태로 표시합니다.
  - 암호 해제 후 다시 업로드하라는 안내를 제공합니다.

- 텍스트 추출 불가
  - 스캔 PDF 가능성을 안내합니다.
  - OCR은 MVP 이후 기능으로 표시할 수 있습니다.

- 임베딩 또는 색인 저장 실패
  - 문서 상태를 `처리 실패`로 표시합니다.
  - 실패 단계와 재시도 버튼을 제공합니다.

### 성공 조건

- 문서가 업로드되고 처리 상태가 추적됩니다.
- 사용 가능한 문서는 검색 색인에 포함됩니다.
- 관리자는 처리 완료 후 테스트 채팅에서 문서 기반 답변을 확인할 수 있습니다.

## 7. 관리자 설정 변경 플로우

### 시작 조건

- 관리자가 로그인되어 있습니다.
- 관리자는 챗봇 설정 수정 권한을 가지고 있습니다.
- 대상 챗봇이 선택되어 있습니다.

### 단계별 흐름

1. 관리자가 챗봇 설정 화면으로 이동합니다.
2. 콘솔은 `GET /api/admin/chatbots/{chatbotId}/settings`로 현재 설정을 불러옵니다.
3. 관리자는 챗봇 이름, 인사말, 위젯 색상, 위치, 운영시간을 수정합니다.
4. 관리자는 페르소나, 답변 톤, 출처 표시 방식, 제한 응답 문구를 수정합니다.
5. 관리자는 개인정보 마스킹 옵션과 빠른 액션 버튼을 수정합니다.
6. 클라이언트는 필수값, 길이, 색상 형식, URL 형식을 검증합니다.
7. 관리자가 저장 버튼을 클릭합니다.
8. 클라이언트는 `PATCH /api/admin/chatbots/{chatbotId}/settings` 요청을 보냅니다.
9. 서버는 권한과 값의 유효성을 검증합니다.
10. 서버는 설정 버전을 증가시키고 변경 이력을 저장합니다.
11. 콘솔은 저장 성공 메시지를 표시합니다.
12. 위젯은 다음 설정 로드 또는 캐시 만료 시 변경된 설정을 반영합니다.

### 예외 상황

- 필수값 누락
  - 저장 전 필드 단위 오류를 표시합니다.

- 잘못된 색상 또는 URL 형식
  - 입력 필드 하단에 검증 오류를 표시합니다.

- 동시 수정 충돌
  - 서버는 설정 버전 충돌을 반환합니다.
  - 콘솔은 최신 설정을 다시 불러오도록 안내합니다.

- 권한 부족
  - 서버는 `403`을 반환합니다.
  - 콘솔은 읽기 전용 상태 또는 접근 불가 안내를 표시합니다.

### 성공 조건

- 설정 변경이 저장되고 변경 이력이 남습니다.
- 관리자 콘솔과 위젯에서 동일한 설정 값이 적용됩니다.
- 잘못된 설정은 저장 전에 차단됩니다.

## 8. 관리자 테스트 채팅 플로우

### 시작 조건

- 관리자가 로그인되어 있습니다.
- 대상 챗봇에 사용 가능한 문서가 하나 이상 있습니다.
- 관리자는 테스트 콘솔 접근 권한을 가지고 있습니다.

### 단계별 흐름

1. 관리자가 테스트 콘솔 화면으로 이동합니다.
2. 콘솔은 현재 챗봇 설정과 문서 색인 상태를 표시합니다.
3. 관리자가 테스트 질문을 입력합니다.
4. 클라이언트는 `POST /api/admin/chatbots/{chatbotId}/test-messages` 요청을 보냅니다.
5. 서버는 실제 사용자 채팅과 동일한 RAG 흐름을 실행합니다.
6. 서버는 답변, 검색된 문서 조각, 점수, 출처, 프롬프트 정책 적용 결과를 반환합니다.
7. 콘솔은 사용자에게 보일 답변 미리보기를 표시합니다.
8. 콘솔은 관리자용 디버그 패널에 검색 문서, 페이지, 조각 내용, 유사도 점수를 표시합니다.
9. 관리자는 답변 품질을 확인하고 필요 시 문서나 설정을 수정합니다.

### 예외 상황

- 사용 가능한 문서 없음
  - 테스트 입력을 비활성화하거나 경고를 표시합니다.
  - 문서 업로드 화면으로 이동하는 액션을 제공합니다.

- 검색 결과 없음
  - 제한 응답 미리보기를 표시합니다.
  - 디버그 패널에는 검색 결과 없음 상태를 표시합니다.

- 모델 호출 실패
  - 사용자용 실패 메시지와 관리자용 요청 ID를 표시합니다.
  - 상세 오류는 서버 로그에서 확인하게 합니다.

- 테스트와 운영 설정 불일치
  - 테스트 콘솔은 현재 저장된 설정 기준으로 실행합니다.
  - 저장되지 않은 설정은 별도 미리보기 모드가 있을 때만 사용합니다.

### 성공 조건

- 관리자는 운영 배포 전 답변 품질을 검증할 수 있습니다.
- 검색 근거와 사용자 표시 답변을 함께 확인할 수 있습니다.
- 테스트 요청은 실제 사용자 로그와 구분되어 저장됩니다.

## 9. 관리자 로그 확인 플로우

### 시작 조건

- 관리자가 로그인되어 있습니다.
- 관리자는 로그 조회 권한을 가지고 있습니다.
- 채팅 또는 시스템 이벤트 로그가 저장되어 있습니다.

### 단계별 흐름

1. 관리자가 로그 화면으로 이동합니다.
2. 콘솔은 기본 기간 필터를 적용해 최근 로그를 조회합니다.
3. 관리자는 기간, 상태, 키워드, 요청 ID, 문서 ID, 세션 ID로 필터링합니다.
4. 클라이언트는 `GET /api/admin/chatbots/{chatbotId}/logs/chat` 요청을 보냅니다.
5. 서버는 권한 범위 내 로그만 반환합니다.
6. 콘솔은 대화 로그, 오류 로그, 문서 처리 로그를 탭 또는 필터로 구분해 표시합니다.
7. 관리자가 특정 로그 행을 클릭합니다.
8. 콘솔은 상세 패널에 질문, 마스킹된 입력, 답변, 출처, 오류 코드, 처리 시간을 표시합니다.
9. 개인정보 원문은 표시하지 않습니다.
10. 필요 시 관리자는 요청 ID를 복사해 개발자 또는 운영자에게 전달합니다.

### 예외 상황

- 로그 없음
  - 빈 상태 메시지와 필터 초기화 버튼을 표시합니다.

- 권한 부족
  - 접근 불가 안내를 표시합니다.

- 개인정보 포함 로그
  - 마스킹된 값만 표시합니다.
  - 원문 복구 기능은 제공하지 않습니다.

- 대량 조회
  - 페이지네이션 또는 커서 기반 조회를 사용합니다.
  - 기간 필터 없이 전체 로그 다운로드는 제한합니다.

### 성공 조건

- 관리자는 채팅 품질과 오류 원인을 추적할 수 있습니다.
- 로그 상세는 개인정보를 노출하지 않습니다.
- 요청 ID로 위젯, API, RAG, 모델 호출 로그를 연결할 수 있습니다.

## 10. 문서 업데이트 후 재색인 플로우

### 시작 조건

- 기존 문서가 이미 업로드되어 사용 중입니다.
- 관리자가 문서 수정 또는 교체 권한을 가지고 있습니다.
- 새 PDF 파일 또는 재처리 요청이 준비되어 있습니다.

### 단계별 흐름

1. 관리자가 문서 관리 화면에서 기존 문서를 선택합니다.
2. 관리자는 `새 버전 업로드` 또는 `재색인` 액션을 선택합니다.
3. 새 버전 업로드인 경우 PDF 파일을 선택합니다.
4. 재색인인 경우 기존 원본 파일을 그대로 사용합니다.
5. 서버는 새 문서 버전 레코드를 생성합니다.
6. 기존 사용 중 버전은 새 버전 처리 완료 전까지 유지됩니다.
7. 문서 처리 워커는 텍스트 추출, 청킹, 임베딩, 색인 저장을 수행합니다.
8. 처리 완료 후 새 색인을 검증합니다.
9. 검증 성공 시 활성 문서 버전을 새 버전으로 전환합니다.
10. 이전 버전의 색인은 비활성화하거나 보관 정책에 따라 삭제 대기 상태로 둡니다.
11. 콘솔은 문서 버전, 활성 상태, 처리 완료 시간을 갱신합니다.
12. 관리자는 테스트 콘솔에서 업데이트된 내용으로 답변되는지 확인합니다.

### 예외 상황

- 새 버전 처리 실패
  - 기존 활성 버전은 계속 사용합니다.
  - 새 버전은 `처리 실패` 상태로 표시합니다.
  - 실패 단계와 재시도 버튼을 제공합니다.

- 재색인 중 사용자 질문 발생
  - 운영 채팅은 기존 활성 색인을 사용합니다.
  - 새 색인은 완료 후 전환됩니다.

- 문서 삭제 요청과 재색인 충돌
  - 서버는 진행 중 작업을 확인하고 충돌 상태를 반환합니다.
  - 콘솔은 먼저 작업 취소 또는 완료 후 삭제하도록 안내합니다.

- 페이지 번호 또는 출처 정보 변경
  - 새 버전 기준 출처를 새로 생성합니다.
  - 이전 로그의 출처는 당시 버전 정보를 유지합니다.

### 성공 조건

- 새 문서 버전이 처리 완료 후 안전하게 활성화됩니다.
- 재색인 실패 시 기존 운영 답변은 영향을 받지 않습니다.
- 테스트 콘솔에서 새 문서 내용이 검색되고 답변에 반영됩니다.
