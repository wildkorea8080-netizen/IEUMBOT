# 18. 배포 및 운영

이 문서는 이음봇(IEUMBOT)의 배포/운영 전략을 정의합니다. 대상은 개발팀과 운영팀이며, 로컬 개발부터 운영 장애 대응까지 공통 실행 기준을 제공합니다.

## 1. 환경 분리 전략

## 1.1 환경 구분

- `local`: 개발자 개인 개발 환경
- `staging`: 운영 전 검증 환경
- `production`: 실제 서비스 환경

## 1.2 분리 원칙

- 환경별 DB, Redis, 파일 스토리지, 도메인 완전 분리
- `production` 데이터의 `local` 직접 반입 금지
- `staging`은 운영과 유사한 설정으로 유지
- 환경별 API 키/시크릿 분리

## 1.3 도메인 예시

- `local`: `http://localhost:*`
- `staging`: `https://staging.ieumbot.example`
- `production`: `https://ieumbot.example`

## 1.4 릴리스 흐름

1. `local` 개발 및 테스트
2. `staging` 배포 및 회귀 검증
3. 운영 승인 후 `production` 배포

## 2. Docker Compose 기반 로컬 실행

## 2.1 로컬 구성 서비스

- `web` (Next.js 관리자 콘솔)
- `api` (FastAPI)
- `worker` (문서 처리 워커)
- `postgres` (PostgreSQL + pgvector)
- `redis`
- `nginx` (옵션)

## 2.2 권장 파일 구조

- `docker-compose.yml` (기본)
- `docker-compose.override.yml` (개발자 로컬 오버라이드)
- `.env` (로컬 전용, 커밋 금지)
- `.env.example` (샘플, 커밋 대상)

## 2.3 로컬 실행 절차(예시)

1. `.env.example` 기반 `.env` 생성
2. 컨테이너 빌드 및 실행
3. DB 마이그레이션 적용
4. 샘플 데이터 로드(선택)
5. 헬스 체크 확인

예시 명령:

```bash
docker compose up -d --build
docker compose exec api alembic upgrade head
docker compose ps
```

## 2.4 로컬 운영 규칙

- OpenAI 키 미설정 시 모델 호출 기능은 제한 모드로 동작 가능해야 함
- 파일 스토리지는 로컬 볼륨 또는 MinIO 사용
- 로그는 콘솔 출력 + 로컬 파일 보관(선택)

## 3. 운영 배포 방식

## 3.1 기본 전략

- 컨테이너 이미지 기반 배포
- 서비스 단위 독립 배포(웹/API/워커)
- 변경 영향 최소화를 위해 점진적 배포

## 3.2 권장 절차

1. 이미지 빌드 및 태깅(`git sha`, 버전 태그)
2. 취약점 스캔(가능 시)
3. 스테이징 배포
4. 스모크 테스트
5. 운영 배포
6. 배포 후 모니터링

## 3.3 배포 단위

- `web`: 관리자 UI
- `api`: REST/SSE 엔드포인트
- `worker`: 인입 작업 처리
- `nginx`: 라우팅/TLS/보안 헤더

## 3.4 배포 승인 기준

- 스테이징 테스트 통과
- 마이그레이션 롤백 가능성 검토
- 변경 영향도 문서화

## 4. Nginx reverse proxy 구조

## 4.1 역할

- TLS 종료
- 라우팅
- 보안 헤더 적용
- 업로드 크기 제한
- SSE 프록시 처리

## 4.2 라우팅 예시

- `/admin` -> `web`
- `/api` -> `api`
- `/widget/*` -> 위젯 정적 파일 또는 위젯 앱
- `/health` -> `api` 헬스 엔드포인트

## 4.3 SSE 운영 포인트

- `proxy_buffering off`
- 타임아웃 기본 API보다 길게 설정
- keep-alive 설정

## 4.4 보안 기본 설정

- HSTS 활성화
- `X-Content-Type-Options: nosniff`
- 적절한 CSP
- 불필요한 메서드 제한

## 5. environment variable 관리

## 5.1 원칙

- 환경 변수는 환경별로 분리
- 브라우저 노출 변수와 서버 전용 변수 분리
- `.env`는 커밋 금지

## 5.2 파일 정책

- 저장소: `.env.example`만 유지
- 실제 값: CI/CD 또는 시크릿 저장소에서 주입

## 5.3 핵심 변수 예시

```text
API_DATABASE_URL=
API_REDIS_URL=
API_OPENAI_API_KEY=
API_SESSION_SECRET=
API_FILE_STORAGE_DRIVER=
API_FILE_STORAGE_BUCKET=
API_ALLOWED_ORIGINS=

WEB_PUBLIC_API_BASE_URL=
WIDGET_PUBLIC_API_BASE_URL=
```

## 5.4 검증

- 애플리케이션 시작 시 필수 변수 존재 검증
- 누락 시 즉시 실패(Fail Fast)

## 6. secret 관리 원칙

- 시크릿은 코드/문서/로그에 저장 금지
- 운영 시크릿은 중앙 비밀 저장소 사용 권장
- 최소 권한 원칙으로 접근 제어
- 키 롤링 절차 문서화

대상 시크릿:

- OpenAI API 키
- DB 접속 비밀번호
- Redis 인증 정보
- 세션 시크릿
- 스토리지 접근 키

## 7. DB migration 운영

## 7.1 원칙

- 스키마 변경은 마이그레이션 도구로 관리(Alembic 권장)
- 수동 SQL 직접 적용 지양
- 마이그레이션과 앱 버전 호환성 유지

## 7.2 배포 절차

1. 배포 전 백업
2. `staging`에 마이그레이션 검증
3. 운영 배포 직전 마이그레이션 적용
4. 헬스 체크 및 주요 기능 확인

## 7.3 주의사항

- 파괴적 변경은 단계적 전환:
  - 컬럼 추가 -> 코드 반영 -> 데이터 이관 -> 구컬럼 제거
- 대용량 테이블 변경은 락/성능 영향 검토

## 8. 백업 및 복구

## 8.1 백업 대상

- PostgreSQL 데이터
- 파일 스토리지 원본(PDF)
- 설정 백업(필요 시)

## 8.2 권장 주기

- DB 전체 백업: 일 1회 이상
- 중요 테이블 증분 백업: 정책에 따라 추가
- 파일 스토리지 백업: 일 1회 이상 또는 버전 기반

## 8.3 보관 정책(초안)

- 일일 백업 7~30일
- 주간 백업 4~8주
- 월간 백업 3~12개월(기관 정책 우선)

## 8.4 복구 목표(초안)

- MVP: `RPO 24시간`, `RTO 4시간`
- 운영 확장: `RPO 1시간`, `RTO 1시간`

## 8.5 복구 점검

- 분기 1회 복구 리허설 권장
- 복구 소요 시간과 실패 원인 기록

## 9. 롤백 전략

## 9.1 애플리케이션 롤백

- 이전 안정 이미지 태그 유지
- 문제 발생 시 `web/api/worker` 서비스별 롤백 가능해야 함

## 9.2 DB 롤백

- 완전 롤백이 어려운 마이그레이션은 사전 복구 계획 필수
- 파괴적 변경은 사전 백업 후 진행

## 9.3 설정 롤백

- 챗봇 설정 버전 관리
- 잘못된 설정은 이전 버전으로 복원 가능해야 함

## 9.4 위젯 롤백

- 정적 자산은 해시/버전 파일명 사용
- 이전 번들 즉시 서빙 가능 상태 유지

## 10. 운영 점검 항목

일일 점검:

- API 오류율
- 채팅 응답 시간(P95)
- 모델 호출 실패율
- 문서 인입 실패 건수
- 큐 적체량
- 디스크/메모리 사용률

주간 점검:

- 실패 로그 상위 원인
- 권한/계정 변경 이력
- 보안 패치 필요 항목
- 백업 정상 여부

배포 후 점검:

- 로그인 동작
- 위젯 설정 로드
- 테스트 채팅 응답
- 문서 업로드 및 인입
- 로그 기록 정상 여부

## 11. 장애 대응 기본 절차

1. 탐지
2. 심각도 분류
3. 영향 범위 확인
4. 임시 완화 조치
5. 근본 원인 분석
6. 영구 조치 반영
7. 사후 보고

심각도 예시:

- `SEV-1`: 전체 서비스 중단
- `SEV-2`: 주요 기능 중단(채팅 또는 업로드)
- `SEV-3`: 일부 기능 저하

필수 운영 규칙:

- 장애 티켓에 `requestId` 또는 `jobId` 포함
- 커뮤니케이션 채널과 책임자 명확화
- 복구 후 재발 방지 액션 아이템 생성

## 12. 문서 교체/재색인 운영 절차

## 12.1 목적

- 서비스 중단 없이 문서 최신화 반영
- 실패 시 기존 문서 기반 답변 유지

## 12.2 절차

1. 운영자가 새 버전 PDF 업로드
2. 시스템이 `document_version` 생성 및 인입 작업 큐 등록
3. 워커가 추출/청킹/임베딩/색인 수행
4. 완료 전까지 기존 활성 버전 유지
5. 처리 성공 시 새 버전 활성화
6. 테스트 채팅으로 반영 확인
7. 필요 시 구버전 비활성/보관 처리

## 12.3 실패 처리

- 인입 실패 시:
  - 새 버전 `failed`
  - 기존 활성 버전 유지
  - 실패 단계/코드 표시
  - 재시도 가능

## 12.4 운영 체크포인트

- 문서 상태 전환 로그 확인
- 출처 페이지/문서명 정상 표시 확인
- 주요 FAQ 질문으로 회귀 테스트 수행

## 13. 운영팀 실행 체크리스트

- 환경별 시크릿 분리 확인
- Nginx SSE 설정 확인
- DB 백업 최근 성공 여부 확인
- 마이그레이션 적용 이력 확인
- 배포 전/후 스모크 테스트 수행
- 장애 대응 연락 체계 최신화
