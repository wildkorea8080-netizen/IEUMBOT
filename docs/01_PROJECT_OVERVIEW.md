# 01. 프로젝트 개요

## 1. 서비스 개요

이음봇(IEUMBOT)은 공공기관, 협회, 기업, 교육기관 등 조직이 보유한 문서를 기반으로 답변하는 문서 기반 AI 챗봇 플랫폼입니다.

- 기존 웹사이트에 플로팅 채팅 위젯을 삽입합니다.
- 사용자는 화면 하단의 플로팅 아이콘을 클릭해 채팅 패널을 엽니다.
- 사용자는 자연어로 질문하고, 챗봇은 업로드된 PDF 문서와 승인된 웹 콘텐츠를 근거로 답변합니다.
- 관리자는 콘솔에서 문서, 챗봇 설정, 말투, 개인정보 마스킹, 빠른 질문, 테스트, 로그를 관리합니다.
- 답변은 추측보다 근거 문서 기반 응답을 우선합니다.

이 문서는 이음봇의 제품 범위, 주요 구성요소, 초기 구현 방향을 개발자, PM, 디자이너가 공유하기 위한 기준 문서입니다.

## 2. 목표

- 조직 웹사이트 방문자가 필요한 정보를 빠르게 찾도록 지원합니다.
- PDF 공지, 지침, 매뉴얼, FAQ, 규정 문서를 AI 답변의 기준 지식으로 사용합니다.
- 운영자가 개발자 도움 없이 문서를 업로드하고 챗봇 동작을 조정할 수 있게 합니다.
- 공공기관과 조직 환경에 필요한 보안, 개인정보 보호, 감사 가능성을 기본 요구사항으로 둡니다.
- 위젯을 기존 사이트에 낮은 변경 비용으로 배포할 수 있게 합니다.

초기 MVP의 목표는 다음 범위를 우선합니다.

- 단일 조직 또는 단일 테넌트 기준의 챗봇 운영
- PDF 업로드 및 문서 기반 검색 응답
- 웹사이트 삽입용 채팅 위젯
- 관리자 콘솔의 기본 설정과 로그 조회
- 개인정보 마스킹 및 기본 접근 제어

## 3. 핵심 가치

- 근거성: 답변은 업로드 문서와 승인된 콘텐츠를 우선 근거로 삼아야 합니다.
- 운영 가능성: 비개발 운영자가 문서와 챗봇 설정을 직접 관리할 수 있어야 합니다.
- 삽입 용이성: 기존 웹사이트에 최소한의 스크립트 또는 설정으로 위젯을 배포할 수 있어야 합니다.
- 안전성: 개인정보, 민감정보, 내부 운영 로그가 노출되지 않도록 설계해야 합니다.
- 추적성: 어떤 문서와 설정이 답변에 영향을 주었는지 확인할 수 있어야 합니다.
- 일관성: 기관의 안내 톤, 명칭, 정책 문구가 답변에 안정적으로 반영되어야 합니다.

## 4. 주요 사용자

- 웹사이트 방문자
  - 기관 웹사이트에서 정책, 신청 절차, 공지, 지원 사업, 이용 방법을 질문합니다.
  - 별도 로그인 없이 공개 정보에 대한 답변을 받는 흐름을 기본으로 합니다.

- 기관 운영자
  - PDF 문서를 업로드하고 문서 상태를 관리합니다.
  - 빠른 질문, 챗봇 이름, 안내 문구, 페르소나, 말투를 설정합니다.
  - 테스트 채팅으로 답변 품질을 확인합니다.

- 관리자
  - 사용자 권한, 조직 설정, 보안 정책, 개인정보 마스킹 옵션을 관리합니다.
  - 채팅 로그, 오류 로그, 사용량 지표를 확인합니다.

- 개발자 또는 웹 담당자
  - 기존 웹사이트에 위젯 스크립트를 삽입합니다.
  - 운영 환경, 도메인 허용 목록, API 키, 배포 설정을 관리합니다.

- PM 및 디자이너
  - 사용자 흐름, 위젯 상태, 관리자 콘솔 화면, 오류 대응 UX를 정의합니다.
  - 답변 신뢰도와 사용 편의성을 기준으로 기능 우선순위를 조정합니다.

## 5. 주요 기능 요약

- 플로팅 채팅 위젯
  - 웹사이트 하단 고정 아이콘으로 표시합니다.
  - 클릭 시 채팅 패널을 열고 닫을 수 있습니다.
  - 모바일과 데스크톱에서 동일한 주요 기능을 제공합니다.

- 문서 기반 질의응답
  - 업로드된 PDF를 파싱하고 검색 가능한 지식으로 변환합니다.
  - 질문과 관련 있는 문서 조각을 검색한 뒤 답변 생성에 사용합니다.
  - 답변에는 가능한 경우 근거 문서명, 페이지, 섹션 정보를 함께 제공합니다.

- 승인 웹 콘텐츠 활용
  - 관리자가 허용한 웹 페이지 또는 도메인 콘텐츠만 지식으로 사용할 수 있습니다.
  - 크롤링 범위, 갱신 주기, 제외 규칙은 별도 명세에서 정의합니다.

- 빠른 질문 및 추천 액션
  - 자주 묻는 질문을 버튼 형태로 제공합니다.
  - 초기 화면, 대화 중, 답변 후 상태에 맞춰 표시할 수 있습니다.

- 개인정보 보호
  - 사용자가 입력한 이름, 전화번호, 이메일, 주민등록번호 패턴 등은 마스킹 대상입니다.
  - 로그 저장 전 마스킹 또는 부분 저장 정책을 적용합니다.

- 오류 및 한계 안내
  - 근거 문서에서 답을 찾지 못한 경우 임의 답변을 피합니다.
  - 담당 부서 문의, 관련 페이지 링크, 추가 질문 유도 등 대체 응답을 제공합니다.

## 6. 관리자 기능 요약

- 문서 관리
  - PDF 업로드, 삭제, 비활성화, 재처리 기능을 제공합니다.
  - 문서별 처리 상태를 표시합니다.
  - 실패 원인과 재시도 액션을 제공해야 합니다.

- 챗봇 설정
  - 챗봇 이름, 인사말, 기본 안내 문구, 위젯 색상, 위치, 아이콘을 설정합니다.
  - 조직별 말투와 금지 표현을 설정할 수 있게 합니다.

- 페르소나 및 톤 설정
  - 공손한 공공기관 안내형 톤을 기본값으로 둡니다.
  - 답변 길이, 문체, 링크 제공 방식, 출처 표시 방식을 조정합니다.

- 개인정보 마스킹
  - 입력값과 로그에 대한 마스킹 규칙을 설정합니다.
  - 기본 탐지 패턴과 조직별 사용자 정의 패턴을 지원할 수 있어야 합니다.

- 빠른 액션 관리
  - 자주 묻는 질문, 바로가기 링크, 문의 안내 버튼을 관리합니다.
  - 표시 순서와 활성 여부를 설정합니다.

- 채팅 테스트
  - 배포 전 관리자 콘솔에서 챗봇 응답을 테스트합니다.
  - 어떤 문서가 검색되었는지 확인할 수 있는 디버그 정보를 제공합니다.

- 로그 및 모니터링
  - 대화 로그, 검색 로그, 오류 로그, 문서 처리 로그를 조회합니다.
  - 개인정보 마스킹 상태와 로그 보관 기간을 표시합니다.

## 7. 시스템 구성 요약

- 웹 위젯
  - 기존 웹사이트에 삽입되는 클라이언트 컴포넌트입니다.
  - 채팅 UI, 세션 식별, 메시지 전송, 답변 렌더링을 담당합니다.

- API 서버
  - 위젯과 관리자 콘솔의 요청을 처리합니다.
  - 인증, 권한, 챗봇 설정, 문서 관리, 채팅 요청을 담당합니다.

- 관리자 콘솔
  - 운영자가 문서와 챗봇 설정을 관리하는 웹 애플리케이션입니다.
  - 테스트 채팅과 로그 조회 화면을 포함합니다.

- 문서 처리 파이프라인
  - PDF 업로드 후 텍스트 추출, 정제, 청킹, 임베딩, 색인 저장을 수행합니다.
  - 처리 상태와 실패 로그를 저장합니다.

- 검색 및 RAG 계층
  - 사용자 질문을 기반으로 관련 문서 조각을 검색합니다.
  - 검색 결과를 모델 입력에 포함해 근거 기반 답변을 생성합니다.

- 데이터 저장소
  - 조직 설정, 사용자 권한, 문서 메타데이터, 채팅 로그, 감사 로그를 저장합니다.
  - 벡터 검색을 위한 임베딩 저장소를 별도로 사용할 수 있습니다.

- 외부 AI 모델
  - 답변 생성, 요약, 검색 질의 보강 등에 사용합니다.
  - 모델 호출 전후에 정책 검사, 마스킹, 로깅을 적용합니다.

## 8. 배포 방식 요약

- 위젯 배포
  - 기존 웹사이트에 스크립트 태그 또는 npm 패키지 형태로 삽입합니다.
  - 조직별 챗봇 ID 또는 공개 설정 키를 사용해 설정을 로드합니다.
  - 허용 도메인 검증을 통해 비인가 사이트 사용을 제한합니다.

- 관리자 콘솔 배포
  - 별도 관리 URL로 제공합니다.
  - 관리자 인증과 역할 기반 권한 제어를 적용합니다.

- 백엔드 배포
  - API 서버, 문서 처리 워커, 검색 저장소, 데이터베이스를 분리 가능한 구성으로 둡니다.
  - MVP에서는 단순한 배포 구성을 우선하되, 워커와 API 서버의 역할은 코드 구조에서 분리합니다.

- 환경 분리
  - 개발, 스테이징, 운영 환경을 분리합니다.
  - 문서 파일, 로그, API 키, 모델 설정은 환경별로 독립 관리합니다.

## 9. 개발 원칙

- 문서 우선
  - 기능 구현 전에 요구사항, API, 데이터 구조, 운영 기준을 문서로 합의합니다.

- 근거 없는 답변 제한
  - 검색된 근거가 부족하면 확정적으로 답하지 않습니다.
  - 모르는 내용은 모른다고 안내하고 대체 경로를 제공합니다.

- 개인정보 최소 수집
  - 서비스 제공에 필요하지 않은 개인정보는 저장하지 않습니다.
  - 로그는 기본적으로 마스킹 후 저장합니다.

- 운영자 중심 설계
  - 반복 운영 작업은 콘솔에서 처리 가능해야 합니다.
  - 실패 상태는 원인과 다음 액션을 함께 보여줍니다.

- 위젯 격리
  - 위젯 스타일과 스크립트는 호스트 웹사이트에 영향을 최소화해야 합니다.
  - 호스트 페이지의 CSS와 충돌하지 않도록 네임스페이스 또는 Shadow DOM 사용을 검토합니다.

- 관측 가능성
  - 채팅 요청, 검색 결과, 모델 호출, 오류를 추적할 수 있어야 합니다.
  - 운영자가 장애와 품질 문제를 재현할 수 있는 최소 로그를 남깁니다.

- 단계적 확장
  - MVP는 단일 조직 또는 제한된 멀티 테넌시를 기준으로 시작합니다.
  - 이후 조직별 권한, 과금, 고급 분석, 다국어 지원으로 확장할 수 있게 경계를 유지합니다.
